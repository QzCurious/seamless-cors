# Direct PAC Replacement Consent Payload

Start planning and start execution use a direct `pacReplacementConsent` payload instead of a generic lifecycle consent requirement list. PAC replacement is the only gateway-owned start confirmation currently modeled, and naming the HTTP and Go shapes after that action keeps the API aligned with the domain language while retaining `consent-required` as the start result kind.

`pacReplacementConsent` is all-or-nothing for the start. The response detail lists all current managed PAC service state, including disabled services with non-empty foreign PAC URLs. The resulting ownership is implied by PAC Replacement Consent and All-Service PAC Management: if start proceeds, the Managed PAC Service Set selected during Gateway Activation is expected to become seamless-cors owned, so surfaces can render transitions such as `foreign -> seamless-cors owned`, `empty -> seamless-cors owned`, or `seamless-cors owned` without an additional result field.

The request input carries only `accepted: true`; clients do not echo the PAC state they saw. `ExecuteStart` guards current state rather than confirming a previous plan: it rechecks current PAC replacement conditions before mutation, proceeds when replacement is no longer needed even if extra consent input is present, and returns fresh `pacReplacementConsent` detail when current state still requires confirmation and acceptance is missing or false.

The same execute-time PAC inspection that rechecks PAC Replacement Consent also selects the Managed PAC Service Set for that Gateway Activation. Gateway Activation fails as an infrastructure error when that selected set is empty, because start cannot deliver Managed System Proxy behavior without any manageable service. Gateway Activation records a non-empty set before CA Ensure and uses it later for scoped PAC install. Scoped PAC install must report the visible requested services it actually updated, and activation fails if that installed service list is empty. During that Gateway Owner run, PAC install, PAC refresh, and Managed PAC Lease checks operate on the recorded set rather than silently expanding to newly discovered services. Newly appearing services are not added to the active owner; they can be included by a later gateway activation. A service from the set disappearing from platform inspection is harmless absence for install, refresh, and lease checking. Lease loss means any visible service in the set has its seamless-cors PAC disabled, replaced, or left on a stale PAC URL. Platform PAC install and refresh use the Managed PAC Service Set as their requested scope: they update visible services in that set, ignore requested services that are currently absent, fail when a visible requested service cannot be updated, and do not touch new unrequested services. Gateway Footprint Cleanup remains marker-based: it inspects current OS service state and removes seamless-cors-owned PAC settings by ownership marker rather than relying on the active owner's service set.

Runtime status exposes the active Managed PAC Service Set when Gateway Runtime is running. This is read-only visibility so users and tests can tell which services are managed in the current owner run; it is not a command surface for changing the set.

Successful Start Guidance Detail includes the managed PAC service names that were actually installed during Gateway Activation, alongside `ManagedPACActive`, so users can see the activated service scope without running status. `ManagedPACActive` remains the semantic active flag; service names are supporting detail rather than the source of truth for active state.

Gateway Facade sorts managed PAC service states and service-name lists by service name before returning surface-neutral command results. This keeps consent details, start guidance, status output, and tests deterministic even when platform service enumeration order varies.

Gateway Activation is implemented as unexported code in `internal/gatewayfacade/activation.go`, not as a separate package. `ExecuteStart` keeps the public facade concurrency guard, including already-running and single-flight mutation handling, then delegates to a package-private activation collaborator that can access `Facade` internals for active runtime publication and watcher setup.

`pacReplacementConsent` is present only on `consent-required` start plan and start result responses. `ready`, `started`, and other non-blocked results omit it so the result kind remains the source of truth for whether user confirmation is required.

This supersedes the generic `LifecycleConsentDetail.Requirements`, `ConsentRequirementKind`, and `acceptedRequirements` shape recorded in ADR 0005.
